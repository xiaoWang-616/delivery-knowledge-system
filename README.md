# 模块项目交付知识库

这是一个专为“按模块交付项目”设计的知识库系统。它不绑定某一个业务项目，而是作为独立的交付调度系统使用：每次接入真实项目路径，先生成项目画像，再整理模块资料包和 design，然后由系统 AI 拆出小任务队列，分段派发给写代码 AI 开发、测试、修复和沉淀规则。

## 核心目标

- 让 AI 写代码前先理解项目结构、接口边界、样式规则和模块依赖。
- 把一个模块拆成可执行、可验收、可复盘的小任务队列。
- 把接口问题、PRD 问题、样式问题、测试问题统一沉淀为问题卡片。
- 把每次踩坑沉淀成规则，让下一个模块少走弯路。
- 支持多个项目复用，只需要为每个项目生成独立项目画像。
- 避免让写代码 AI 一次性生成大量代码，降低跑偏和上下文混乱风险。

## 当前阶段

第一期已经具备 Markdown/Obsidian 知识库和可视化控制台。控制台可以创建一次性资料包、展示执行卡片、生成 Markdown，并通过本地 runner 只读扫描真实项目路径、生成 AI 上下文包、生成 AI 交付执行包、生成 design + 小任务队列、受控执行 `typecheck` / `lint` / `build`、写入项目实例知识库、生成模块依赖图、步骤卡片、测试用例和验收报告。

后续再逐步增加：

- 可视化看板与执行记录持久化。
- 更完整的页面自动点测、截图对比和接口验收回填。
- AI 自动开发、测试、修复和知识沉淀。

## 目录说明

- `00-系统说明`：系统定位、使用流程、阻塞分级。
- `01-项目接入`：项目画像、项目扫描、规则提取模板。
- `02-模块交付`：模块资料包、依赖图、design、小任务队列、开发步骤卡片。
- `03-测试验收`：功能测试、样式测试、接口测试、验收报告。
- `04-问题复盘`：问题卡片、二次修复、规则沉淀。
- `05-AI提示词`：项目分析、模块开发、代码审查、二次修复提示词。
- `06-样板模块`：真实项目模块样例，用来展示模板如何填写。
- `07-使用手册`：给实际使用者看的启动说明、模块启动流程。
- `08-项目实例`：每个真实项目的执行工作区模板和项目记录。
- `09-模块看板`：模块状态、依赖、开发进度、问题追踪的可视化模板。
- `10-测试修复闭环`：一轮开发后的测试、问题汇总、人工确认和二次修复流程。
- `11-规则库`：接口、样式、性能、AI 写代码和项目差异的通用规则。
- `apps/delivery-console`：自动交付控制台前端，用来创建任务包、查看执行进度、生成交付结果。

## 使用方式

1. 用 Obsidian 打开本目录。
2. 先阅读 `07-使用手册/从0开始使用.md`。
3. 启动新模块时，把 `05-AI提示词/新模块启动总提示词.md` 复制给 AI。
4. 接入真实项目时，复制 `08-项目实例/项目实例模板`，创建项目自己的工作区。
5. 接入项目时，先填写项目画像和项目实例里的 `项目总览.md`。
6. AI 开发前必须先阅读 `11-规则库`。
7. 开发模块前，先填写模块资料包和模块依赖图。
8. 为模块创建模块状态卡片、依赖看板、开发进度看板和问题追踪看板。
9. AI 开发前必须先阅读项目画像、模块资料包、开发步骤卡片。
10. 第一次测试后，按 `10-测试修复闭环/测试执行说明.md` 执行验收。
11. 把问题写入问题卡片或问题追踪看板，并汇总到问题清单。
12. 人工确认问题后，用二次修复提示词启动修复。
13. 修复完成后，把可复用经验沉淀为规则库或项目特有规则。
14. 提交前阅读 `07-使用手册/提交前检查.md`，区分系统代码、可复用知识和本地运行产物。

## 可视化控制台

V1 前端入口：

```text
apps/delivery-console
```

它用于把“看和沉淀”的 Markdown 知识库，连接到“操作和交付”的可视化流程。

当前控制台已经支持任务包、系统自检、资料依赖阶段卡、执行计划、状态卡片、风险归档、Markdown 生成、真实项目只读扫描、AI 上下文包生成、AI 交付执行包生成、设计与任务队列生成、当前任务派发、写代码 AI 报告回填、轻量 review、修复任务生成、总验收与规则沉淀、受控命令验收、运行记录持久化、历史运行列表、可视化模块依赖图、步骤卡片、问题追踪、测试用例、验收报告和 Obsidian 项目实例写入。真实业务项目写入、页面自动点测和 AI 自动调用调度放到 V2/V3。

推荐在知识库根目录一键启动：

```bash
bash scripts/start-console.sh
```

启动前可先检查本机环境、依赖、端口和 runner health：

```bash
bash scripts/check-console.sh
```

停止前端和 runner：

```bash
bash scripts/stop-console.sh
```

如果只想查看会停止哪些进程：

```bash
bash scripts/stop-console.sh --dry-run
```

- 前端地址：`http://localhost:5174`
- 本地 runner：`http://localhost:5176`
- 前端项目目录：`apps/delivery-console`

如果需要手动启动，也可以进入 `apps/delivery-console` 后分别运行 `pnpm run dev` 和 `pnpm run runner`。

runner 当前只写本知识库目录下的 `08-项目实例`，不会修改你填写的真实项目路径。点击“执行命令验收”时，runner 只会读取目标项目 `package.json` 中已有的 `typecheck`、`lint`、`build` 脚本并按顺序执行；不会接受任意命令。

点击“系统自检”时，前端会请求 runner 的 `/api/health`，检查 runner 是否在线、知识库根目录是否正确、`08-项目实例`、`11-规则库`、控制台目录和 `package.json` 是否存在。命令行也可以运行：

```bash
bash scripts/check-console.sh
```

点击“生成 AI 上下文包”时，runner 会读取任务包中的本地接口文档、Demo、旧项目参考和 PRD 路径，生成 `模块上下文包.auto.md`、`接口能力矩阵.auto.md`、`样式参考摘要.auto.md`、`旧项目风险点.auto.md` 和 `AI执行提示词.auto.md`。这一步用于把资料压成 AI 可消费的上下文，前端只负责触发和展示。

点击“生成 AI 交付执行包”时，runner 会在 `08-项目实例/{项目名}/{模块名}/执行包/` 下生成 `execution-manifest.auto.json`、`execution-plan.auto.md`、`ai-run-prompt.auto.md`、`acceptance-checklist.auto.md` 和 `unresolved-questions.auto.md`。这一步用于把上下文和规则整理成系统 AI 的调度输入，不推荐把它直接交给写代码 AI 一次性完成整个模块。

点击“生成设计与任务队列”时，runner 会在 `08-项目实例/{项目名}/{模块名}/设计与任务/` 下生成系统 AI 调度写代码 AI 所需的分段任务文件：

```text
module-design.auto.md
task-queue.auto.json
task-01.prompt.auto.md
task-02.prompt.auto.md
review-checklist.auto.md
progress.auto.md
```

写代码 AI 每次只接收一个 `task-xx.prompt.auto.md`，完成后由系统 AI 做轻量 review，再决定继续下一个任务还是生成修复任务。

点击“复制当前任务 prompt”时，runner 会把当前可执行任务标记为 `assigned`，返回并复制对应 `task-xx.prompt.auto.md`。写代码 AI 完成后，把报告贴回“写代码 AI 完成报告”，点击“提交系统 review”。runner 会把 review 写入 `设计与任务/reviews/`，通过则推进到下一个任务，不通过则生成 `task-xx-fix.prompt.auto.md`。

点击“恢复任务队列”时，runner 会按当前项目名和模块名读取：

```text
08-项目实例/{项目名}/{模块名}/设计与任务/task-queue.auto.json
```

这用于刷新页面、换浏览器或换人接手后继续任务，不强依赖运行记录。恢复成功后，前端会重新显示当前任务、已完成数和后续派发状态。

点击“生成总验收”时，runner 会读取 `task-queue.auto.json`、`reviews/`、命令验收结果和知识库写入结果，生成：

```text
08-项目实例/{项目名}/{模块名}/设计与任务/final-acceptance.auto.md
```

总验收状态分为 `success`、`warning`、`blocked`、`error`。`success` 表示任务、review、命令验收和知识沉淀都已具备；`warning` 表示仍有待处理项；`blocked` 表示存在阻断任务。

运行记录会保存到：

```text
08-项目实例/{项目名}/{模块名}/runs/{runId}.json
08-项目实例/{项目名}/{模块名}/runs/latest.json
```

`latest.json` 用于控制台恢复最近一次运行，带 `runId` 的文件用于历史追溯。

控制台的“历史运行”会读取同一目录下的历史 run 文件，支持恢复指定记录；“模块依赖图”会把项目路径、项目画像、接口文档、Demo、旧项目、模块计划、命令验收和知识沉淀串成节点，帮助人确认 AI 后续消费上下文的顺序。
