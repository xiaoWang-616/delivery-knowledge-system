# 性能规则

## 强规则

### 1. 禁止整页刷新解决局部数据更新

写操作成功后，不允许使用：

```text
window.location.reload()
location.reload()
重新跳转当前页面来刷新
```

正确做法：

- 刷新相关 query。
- 更新局部状态。
- 只重新请求受影响列表或详情。

### 2. 搜索不能每个字符都请求

搜索输入必须考虑：

- debounce。
- 手动点击查询。
- 回车查询。
- 中文输入法组合输入。

不能出现输入一个字母就失焦或触发大量请求。

### 3. 列表不能全量拉取后前端过滤

面向企业级后台时，默认认为数据会持续增长。

禁止：

- 默认拉 1000 条、10000 条再过滤。
- 前端本地分页替代服务端分页。
- 用大 pageSize 掩盖接口分页问题。

### 4. 避免重复请求不可解释

页面首次加载、筛选、分页、进入详情时，请求数量必须可解释。

如果出现重复请求，要检查：

- query key 是否稳定。
- 组件是否重复挂载。
- effect 依赖是否错误。
- 是否同时有旧脚本和新代码在请求。

## 建议规则

### 1. 详情数据按需加载

详情页可以分块加载：

- 主体信息优先。
- tab 内容按激活加载。
- 附件、日志、统计等重数据延后加载。

### 2. 字典数据可缓存

省市区、业务范围、状态枚举、角色列表等相对稳定数据可以缓存，但要有过期或刷新策略。

### 3. 表格列多时考虑布局

列很多的后台表格，要考虑：

- 横向滚动。
- 固定操作列。
- 文案省略和 tooltip。
- 列宽稳定。

## 项目特有规则

项目特有性能规则写在项目实例目录，例如：

```text
08-项目实例/项目名/项目性能规则.md
```

适合记录：

- 哪些接口慢。
- 哪些页面数据量大。
- 哪些字典可以缓存。
- 哪些模块需要按需加载。
