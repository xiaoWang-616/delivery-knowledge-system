# 项目差异规则

## 为什么需要项目差异规则

这个知识库是跨项目复用的，但每个项目都有自己的：

- 技术栈。
- 目录结构。
- 组件库。
- 样式风格。
- 接口约定。
- 提交流程。
- 测试命令。

所以规则库不能要求所有项目都长成一样。通用规则给边界，项目差异规则给落地方式。

## 强规则

### 1. 先识别项目现状，再套规则

接入新项目时，AI 必须先扫描：

- `package.json`
- 路由入口。
- 接口封装目录。
- 组件目录。
- 样式入口。
- 测试命令。
- 项目内 AI 规则文档。

然后再决定怎么写代码。

### 2. 项目已有规范优先于 AI 个人偏好

如果项目已有成熟结构，AI 应该延续，不应强行替换。

例如：

- 项目用 React Query，就继续用 query/mutation。
- 项目用 Zustand，就继续当前 store 风格。
- 项目已有组件库，就优先复用。
- 项目已有 API client，就不要另起 axios 实例。

### 3. 项目特有规则不能违反通用强规则

项目特有规则可以补充通用规则，但不能绕过强规则。

如果发生冲突，例如项目旧代码大量使用整页 reload，但规则库禁止整页刷新，必须人工确认，而不是照搬旧项目。

## 建议规则

### 1. 为每个项目建立项目实例

每个真实项目都应复制：

```text
08-项目实例/项目实例模板
```

并改名为项目名。

项目特有资料、模块清单、风险、看板都放在项目实例目录。

### 2. 项目差异写成显式规则

项目特殊点不要只留在聊天中。

应该写成：

```text
08-项目实例/项目名/项目差异规则.md
```

建议格式：

| 规则 | 原因 | 影响范围 | 是否强制 |
| --- | --- | --- | --- |
|  |  |  |  |

### 3. 样板模块只作为参考

`06-样板模块` 里的内容是示例，不是所有项目的强制实现。

使用时应该参考：

- 怎么整理资料。
- 怎么拆步骤。
- 怎么记录问题。
- 怎么沉淀规则。

不应该照搬具体业务字段。
