# 接口规则

## 强规则

### 1. 不猜接口字段

如果接口文档没有定义某个请求参数、响应字段或枚举值，前端不能自行添加。

错误做法：

```text
页面要按区筛选，但接口没有 districtCode，于是前端自己传 districtCode。
```

正确做法：

```text
记录为接口能力缺口。
如果不影响主流程，可做 UI 降级或提示。
如果影响核心业务，列为 P0，等待后端或产品确认。
```

### 2. 接口真相优先级

接口优先级固定为：

```text
OpenAPI/proto/接口手册 > 后端代码确认 > 当前项目接口封装 > 旧项目代码
```

旧项目代码不能作为接口真相，尤其不能照搬旧项目里的：

- mock 数据。
- 桥接脚本。
- 本地过滤参数。
- 历史兼容字段。
- 临时测试接口。

### 3. 页面组件不能直接请求接口

企业级项目里，页面和组件不要直接 `fetch` 或拼接口路径。

推荐分层：

```text
features/*/api      接口函数
features/*/model    类型、枚举、映射
features/*/hooks    query/mutation 和业务状态
features/*/pages    页面组合
features/*/components 业务组件
```

如果项目已有自己的结构，以项目结构为准，但原则不变：接口要集中封装。

### 4. 列表必须优先服务端分页

列表页默认使用服务端分页。

禁止为了省事：

- 一次拉大量数据。
- 前端本地搜索。
- 前端本地分页。
- 用很大的 `pageSize` 假装分页。

除非接口明确只返回小型字典数据。

### 5. 写操作错误必须展示

新增、编辑、删除、停用、审核、转派等写操作：

- 成功要有反馈。
- 失败要展示后端错误信息。
- 不能静默失败。
- 不能吞掉错误后刷新页面。

## 建议规则

### 1. 请求参数集中构造

筛选参数、分页参数、排序参数建议集中构造，避免页面里到处拼对象。

### 2. 响应字段做兼容映射

接口字段可能出现 `null`、空数组、字符串数字等情况。类型和展示层要做基础兼容，避免页面白屏。

### 3. 字典和枚举集中维护

状态、业务范围、角色、权限等枚举，不要散在页面里写魔法字符串。

## 项目特有规则

项目特有接口规则写在项目实例目录，例如：

```text
08-项目实例/项目名/项目接口规则.md
```

如果项目特有规则允许某些特殊兼容，必须写清楚：

- 为什么特殊。
- 影响哪些模块。
- 什么时候可以删除。
