# 从 0 开始使用

这份说明给不会前端、但需要推动 AI 开发模块的人使用。

你不需要先理解前端工程细节，只需要把真实项目资料交给 AI，并要求 AI 按本知识库的流程先分析、再开发、再测试、再沉淀。

## 1. 打开知识库

如果使用 Obsidian：

1. 打开 Obsidian。
2. 选择 Open folder as vault。
3. 选择目录：

```text
/Users/wangxiaoyu/Documents/delivery-knowledge-system
```

如果暂时没有安装 Obsidian，也可以直接用编辑器打开这个目录，Markdown 文件都可以正常阅读。

## 2. 每次开发前要准备什么

至少准备这些资料：

| 资料 | 是否必需 | 说明 |
| --- | --- | --- |
| 真实项目路径 | 必需 | AI 要进入这个项目读代码和写代码 |
| 模块名称 | 必需 | 例如代理商管理、客户库、商机管理 |
| 接口文档 | 必需 | 接口路径、参数、响应字段以它为准 |
| 设计图或 Demo | 强烈建议 | 样式和交互优先参考它 |
| 旧项目路径 | 可选 | 只能参考样式和交互，不能当接口真相 |
| PRD 或需求说明 | 可选 | 用来补业务规则和验收标准 |

资料不全也可以开始，但 AI 必须把缺失内容记录成问题，不能擅自猜业务规则或接口字段。

## 3. 正确启动方式

不要直接说：

```text
帮我写客户库页面。
```

应该说：

```text
请先阅读 /Users/wangxiaoyu/Documents/delivery-knowledge-system/README.md
并按 /Users/wangxiaoyu/Documents/delivery-knowledge-system/05-AI提示词/新模块启动总提示词.md 执行。

真实项目路径：
...

模块名称：
...

接口文档：
...

设计图或 Demo：
...

旧项目参考：
...

先输出项目分析和模块开发方案，不要直接写代码。
```

## 4. AI 第一轮应该产出什么

AI 第一轮不应该直接改代码，应该先输出：

- 项目结构分析
- 样式和组件规则
- 接口来源和优先级
- 已读取的规则库结论
- 模块范围
- 模块依赖
- 接口能力矩阵
- 开发步骤卡片
- P0/P1/P2 风险
- 需要人工确认的问题
- 验收方式

你确认方案后，AI 才进入写代码。

## 5. 写代码时怎么控制不混乱

一个模块不要一次性全写完，按步骤推进：

1. 列表页
2. 详情页
3. 新增/编辑等写操作
4. 子资源或依赖能力
5. 样式对齐和测试验收

每完成一步，都要让 AI 告诉你：

- 改了哪些文件
- 实现了哪些功能
- 哪些接口已联调
- 哪些问题没有解决
- 怎么验收

## 6. 遇到问题怎么办

不要让 AI 直接靠猜继续写。

按阻塞等级处理：

| 等级 | 处理方式 |
| --- | --- |
| P0 | 停下，必须人工确认 |
| P1 | 继续开发，但记录到问题卡片 |
| P2 | 继续开发，测试阶段统一处理 |

典型 P0：

- 接口文档没有核心接口。
- 写操作字段不明确。
- 权限、状态流转、金额、删除类逻辑不明确。

典型 P1：

- 页面有三级筛选，但接口只支持一级筛选。
- Demo 有字段，但接口暂时没有返回。
- 测试环境缺少特定数据。

## 7. 第一次写完后必须做什么

第一次写完不是结束，必须进入测试和复盘：

1. 跑类型检查、lint、build。
2. 打开页面点测核心流程。
3. 对照 Demo 或设计图检查样式。
4. 检查 Network，确认没有无效接口、重复接口、整页 reload。
5. 把问题写到问题卡片。
6. 人工确认问题后，再启动二次修复。

## 8. 每个模块结束后要沉淀什么

模块结束后，至少沉淀三类东西：

- 本模块资料包。
- 本模块问题卡片。
- 可复用的规则。

如果一个问题以后还会遇到，就不要只留在聊天记录里，要写进规则库或样板模块。

规则库位置：

```text
11-规则库
```

项目特有规则建议写在项目实例目录，例如：

```text
08-项目实例/项目名/项目差异规则.md
```
