# 新模块启动流程

这份流程用于启动一个新的业务模块开发。

目标是让 AI 先把项目和模块分析清楚，再进入代码实现，避免一开始就写出一堆不可控代码。

## 阶段 1：输入资料

启动前，把下面信息交给 AI：

| 信息 | 示例 | 必填 |
| --- | --- | --- |
| 知识库路径 | `/Users/wangxiaoyu/Documents/delivery-knowledge-system` | 是 |
| 真实项目路径 | `/Users/wangxiaoyu/Desktop/object/gil-business-web` | 是 |
| 模块名称 | 代理商管理 | 是 |
| 接口文档路径 | `/Users/wangxiaoyu/Desktop/object/gil-business-jiekou/接口文档.md` | 是 |
| Demo/设计图路径 | `/Users/wangxiaoyu/Downloads/Demo/...html` | 建议 |
| 旧项目路径 | `/Users/wangxiaoyu/Desktop/object/gil-business-web-old/gil-business-web` | 可选 |
| PRD 路径 | 飞书文档、本地 Markdown、截图 | 可选 |

## 阶段 2：项目分析

AI 必须先阅读：

- `README.md`
- `00-系统说明/使用流程.md`
- `00-系统说明/阻塞分级规则.md`
- `11-规则库/README.md`
- `11-规则库/接口规则.md`
- `11-规则库/样式规则.md`
- `11-规则库/性能规则.md`
- `11-规则库/AI写代码规则.md`
- `11-规则库/项目差异规则.md`
- `08-项目实例/项目实例模板/README.md`
- `01-项目接入/项目画像模板.md`
- `01-项目接入/样式组件接口分析模板.md`

然后输出：

- 当前项目技术栈。
- 路由结构。
- 页面目录结构。
- 接口封装位置。
- 组件复用方式。
- 样式来源和主题规则。
- 已存在的性能约束。
- 通用规则库和项目特有规则的冲突点。

接入真实项目时，建议先复制：

```text
08-项目实例/项目实例模板
```

并改名为真实项目名，例如：

```text
08-项目实例/gil-business-web
```

后续项目级资料、模块清单、风险清单和交付看板都记录在这个项目实例目录里。

如果项目里已有 AI 规则文档，也必须一起阅读，例如：

```text
docs/ai-frontend-guide.md
docs/ai-coding-guide.md
```

## 阶段 3：模块资料包

AI 根据模板整理模块资料包：

- `02-模块交付/模块资料包模板.md`
- `02-模块交付/模块依赖图模板.md`
- `02-模块交付/接口能力矩阵模板.md`
- `02-模块交付/开发步骤卡片模板.md`
- `09-模块看板/模块状态卡片模板.md`
- `09-模块看板/模块依赖看板模板.md`
- `09-模块看板/开发进度看板模板.md`
- `09-模块看板/问题追踪看板模板.md`

输出内容必须包含：

- 模块页面范围。
- 功能范围。
- 不做的范围。
- 依赖接口。
- 依赖页面。
- 依赖数据。
- 样式优先级。
- 风险点。
- 开发步骤。
- 当前状态卡片。
- 模块依赖看板。
- 开发进度看板。
- 问题追踪看板。

## 阶段 4：人工确认

AI 输出方案后，人工只需要确认这些问题：

- 模块范围是否正确。
- 样式参考是否正确。
- 接口文档是否是最新。
- 哪些 P0 问题必须先问产品或后端。
- 哪些 P1 问题可以先记录，后面统一处理。

如果方案不清楚，不要让 AI 写代码。

## 阶段 5：分步开发

推荐顺序：

| 步骤 | 内容 | 验收重点 |
| --- | --- | --- |
| 1 | 列表页 | 搜索、筛选、分页、空态、loading |
| 2 | 详情页 | 路由返回、刷新恢复、只读信息 |
| 3 | 写操作 | 新增、编辑、状态变更、局部刷新 |
| 4 | 子资源 | 成员、附件、授权、配置等 |
| 5 | 样式和测试 | 对齐 Demo，清理问题 |

每一步完成后都要输出：

- 文件变更。
- 功能说明。
- 接口说明。
- 测试结果。
- 剩余问题。

## 阶段 6：测试验收

AI 必须按模板生成测试结果：

- `10-测试修复闭环/测试执行说明.md`
- `10-测试修复闭环/问题汇总模板.md`
- `03-测试验收/测试用例模板.md`
- `03-测试验收/样式验收模板.md`
- `03-测试验收/验收报告模板.md`

测试至少覆盖：

- 正常数据。
- 空数据。
- 接口失败。
- 搜索筛选。
- 分页。
- 新增/编辑后局部刷新。
- 浏览器返回。
- 页面刷新恢复。
- 样式对齐。
- 性能风险。

## 阶段 7：问题卡片和二次修复

第一次验收发现的问题，不要散在聊天里。

要统一写成：

- `04-问题复盘/问题卡片模板.md`
- `04-问题复盘/二次修复模板.md`
- `10-测试修复闭环/问题汇总模板.md`
- `10-测试修复闭环/人工确认清单.md`
- `10-测试修复闭环/二次修复启动提示词.md`

问题需要标明：

- 问题类型。
- 影响范围。
- 证据。
- 阻塞等级。
- 责任方。
- 修复建议。
- 是否沉淀为规则。

## 阶段 8：规则沉淀

模块结束后，把能复用的经验沉淀到：

- `04-问题复盘/规则沉淀模板.md`
- `11-规则库`
- 项目实例目录下的项目特有规则。

沉淀判断标准：

- 这个坑以后还会遇到。
- 这个规则能减少 AI 犯错。
- 这个规则能帮助不会前端的人判断方向。
