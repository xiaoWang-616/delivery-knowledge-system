# 阻塞分级规则

开发过程中遇到问题时，不应该一律停下，也不应该一律继续。必须按影响范围分级处理。

## P0：必须停下来人工确认

遇到以下问题必须暂停当前实现，先让人确认：

- 接口文档没有对应接口，但功能强依赖该接口。
- 写操作参数不明确，可能写错数据。
- 状态流转规则不明确，例如审核、停用、作废、转派。
- 删除规则不明确，例如物理删除还是软删除。
- 权限规则不明确。
- 财务、合同、金额、客户归属等高风险字段不明确。
- 接口文档和 PRD 冲突，且会影响业务结果。
- 继续写会导致错误数据进入后端。

处理方式：

1. 记录 P0 问题卡。
2. 写清楚阻塞原因。
3. 写清楚需要谁确认。
4. 等确认后再继续。

## P1：可以继续，但必须记录风险

遇到以下问题可以继续开发，但必须写入问题卡：

- 某个筛选接口暂不支持。
- 排序字段不可靠。
- 某个非核心展示字段缺失。
- 枚举中文映射不完整。
- 统计字段暂时没有。
- 下载、上传等功能缺少真实测试数据。
- 接口分页实现不稳定，但页面可以先按契约写。

处理方式：

1. 继续实现已有能力。
2. 不猜接口文档没有的字段。
3. 必要时做降级展示或提示。
4. 在测试报告中明确标注风险。

## P2：继续开发，验收阶段统一优化

遇到以下问题不用打断开发：

- 文案细节。
- 表格列宽微调。
- 字号、边距、颜色微调。
- tooltip 展示字段可优化。
- 空态文案可优化。
- loading 样式可优化。
- 非核心交互动效。

处理方式：

1. 继续开发。
2. 在验收阶段统一整理。
3. 如果反复出现，则沉淀为样式或组件规则。

## 判断口诀

- 会写错数据：停。
- 会造成状态错乱：停。
- 会影响权限、财务、删除：停。
- 只是缺少展示能力：记录，继续。
- 只是样式细节：继续，验收时修。
