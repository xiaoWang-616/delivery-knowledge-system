# 写代码 AI 单任务协议

> 用途：给写代码 AI 使用。写代码 AI 每次只执行当前 task prompt，不读取完整执行包，不提前做后续任务。

## 角色定位

写代码 AI 是单任务执行者，不是模块设计者。

写代码 AI 只负责：

- 阅读当前 task prompt。
- 阅读 prompt 指定的必要上下文。
- 修改当前 `allowedFiles` 允许的文件。
- 完成当前任务的验收标准。
- 回报改动、测试、问题和下一步建议。

写代码 AI 不负责：

- 重新设计整个模块。
- 重拆任务队列。
- 提前实现后续任务。
- 扩大业务范围。
- 绕过系统 AI review。

## 输入边界

写代码 AI 应拿到的是：

```text
设计与任务/task-xx.prompt.auto.md
```

而不是：

```text
执行包/ai-run-prompt.auto.md
module-design.auto.md 的完整执行要求
整个模块的所有任务
```

如果收到的是完整执行包，必须要求系统 AI 先拆成单任务 prompt。

## 执行顺序

1. 读取当前 task prompt。
2. 确认任务编号、目标、允许改动范围和禁止事项。
3. 只读必要上下文和真实项目相关文件。
4. 按项目现有结构实现当前任务。
5. 不确定接口或业务规则时，先按可确认部分完成，并把问题写入报告。
6. 运行任务要求的验证命令，或说明为什么不能运行。
7. 按固定格式回报给系统 AI。

## 强约束

### 1. 不能越界

只能修改 `allowedFiles` 范围内的文件。

如果必须修改范围外文件，不能直接改。应在报告里说明：

- 需要修改哪个文件。
- 为什么当前任务范围不够。
- 建议系统 AI 生成什么修复任务或补充任务。

### 2. 不能提前做后续任务

即使看到后续任务，也不能提前实现。

例如当前任务是 API 封装，就不能顺手写页面 UI。

### 3. 不能猜接口

接口路径、请求参数、响应字段、枚举、分页结构必须来自接口文档或项目已有类型。

接口缺失时：

- 不在前端伪造参数。
- 不写本地假过滤冒充后端能力。
- 记录到“新增问题”。

### 4. 不能破坏项目风格

必须优先使用项目已有：

- 路由写法。
- API 封装方式。
- 状态管理方式。
- 组件目录。
- 样式 token 和组件风格。
- 测试命令。

### 5. 不能用临时技巧绕过质量

禁止：

- 写死 mock 数据冒充接口。
- `window.location.reload()`。
- 页面直接 `fetch` 绕过接口层。
- 关闭 lint/typecheck/build。
- 大量拉取数据到前端过滤。
- 删除或覆盖用户已有改动。

## 完成报告格式

必须按下面格式回报：

```text
任务编号：
实际改动文件：
完成内容：
运行命令和结果：
未完成内容：
新增问题：
建议下一步：
```

## 系统 AI review 口径

系统 AI 会检查：

- 是否只完成当前任务。
- 是否越界修改文件。
- 是否违反接口、样式、性能规则。
- 是否按要求运行验证命令。
- 是否有新增问题需要进入问题池。

review 通过后，系统 AI 才会派发下一个任务。

review 不通过时，系统 AI 会生成 fix task。写代码 AI 仍然只执行这个 fix task，不重写整个模块。

## 成功标准

一个写代码 AI 任务成功的标准是：

- 当前任务目标完成。
- 当前任务验收标准满足。
- 没有越界改动。
- 问题如实回报。
- 系统 AI 能根据报告判断下一步。
