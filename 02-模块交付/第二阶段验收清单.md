# 第二阶段验收清单

> 用途：验收“系统 AI / 写代码 AI 闭环”是否已经可用。第二阶段的目标不是自动写完整项目，而是让系统 AI 能稳定做 design、拆任务、派任务、review、修复和沉淀。

## 阶段定位

第二阶段解决的是调度闭环：

```text
一次性给齐资料
  ↓
系统 AI 生成 design 和任务队列
  ↓
系统 AI 每次派发一个 task prompt
  ↓
写代码 AI 只执行当前 task
  ↓
系统 AI review 报告和 git 改动范围
  ↓
通过则继续，不通过则生成修复任务
  ↓
总验收和知识沉淀
```

前端控制台只做展示、录入和触发。复杂判断仍属于系统 AI 和 runner。

## 必须具备

| 能力 | 验收口径 | 状态 |
| --- | --- | --- |
| 一次性资料包 | 可以录入项目路径、模块目标、接口文档、Demo、旧项目、PRD 和约束规则。 | 已具备 |
| 项目扫描 | runner 能只读扫描真实项目，生成项目画像，不修改真实项目。 | 已具备 |
| 上下文包 | 能生成模块上下文包、接口能力矩阵、样式摘要和旧项目风险点。 | 已具备 |
| 执行包 | 能生成 execution manifest、执行计划、总提示词、验收清单和待确认问题。 | 已具备 |
| design 和任务队列 | 能生成 `module-design.auto.md`、`task-queue.auto.json` 和单任务 prompt。 | 已具备 |
| 系统 AI 协议 | 有明确的系统 AI 调度协议，规定 design、拆任务、派发、review、修复和沉淀。 | 已具备 |
| 写代码 AI 协议 | 有明确的单任务协议，规定写代码 AI 只做当前 task，不越界。 | 已具备 |
| 单任务派发 | 每次只派发当前可执行 task，派发时记录目标项目 git baseline。 | 已具备 |
| 完成报告回填 | 写代码 AI 能按固定格式回填改动、测试、问题和下一步建议。 | 已具备 |
| 系统 review | runner 能检查报告内容和本任务新增 git 改动是否超出 `allowedFiles`。 | 已具备 |
| 修复任务 | review 不通过或问题池问题可以生成独立 fix task。 | 已具备 |
| 问题池 | 能汇总资料缺口、review 问题、命令验收问题、知识写入问题和总验收问题。 | 已具备 |
| 命令验收 | 能受控执行目标项目已有的 `typecheck`、`lint`、`build` 脚本。 | 已具备 |
| 知识沉淀 | 能把项目画像、资料清单、模块资料包、依赖图、步骤卡片、问题追踪、测试用例和验收报告写入知识库。 | 已具备 |
| 运行记录 | 能保存、恢复最近运行和历史运行。 | 已具备 |
| 总验收 | 能根据任务、review、命令验收、知识沉淀和问题池生成最终状态。 | 已具备 |

## 手工验收流程

1. 在控制台填写一个模块资料包，至少包含真实项目路径、模块名称和需求说明。
2. 点击“系统自检”，确认 runner、知识库根目录和关键目录正常。
3. 点击“扫描项目”，确认生成项目画像，且真实项目没有新增文件。
4. 点击“生成 AI 上下文包”，确认模块目录出现上下文、接口矩阵、样式摘要和风险文件。
5. 点击“生成 AI 交付执行包”，确认出现执行包目录和待确认问题。
6. 点击“生成设计与任务队列”，确认出现 `设计与任务/task-queue.auto.json` 和多个 `task-xx.prompt.auto.md`。
7. 点击“复制当前任务 prompt”，确认当前任务变为 `assigned`，prompt 中包含 `写代码AI单任务协议.md`。
8. 模拟或粘贴写代码 AI 完成报告，点击“提交系统 review”。
9. 如果报告合格且 git 改动范围合格，确认任务推进到 `done`。
10. 如果报告不合格或越界，确认系统生成 `task-xx-fix.prompt.auto.md`，并把修复任务置为当前优先任务。
11. 在问题池中选择一个问题生成修复任务，确认不会重复生成同一个未完成修复任务。
12. 点击“执行命令验收”，确认只运行目标项目 `package.json` 已有的受控脚本。
13. 点击“写入知识库”，确认只写入本知识库 `08-项目实例`。
14. 点击“生成总验收”，确认状态能区分 `success`、`warning`、`blocked`、`error`。
15. 刷新页面后点击“恢复任务队列”或恢复历史运行，确认能继续当前模块。

## 不属于第二阶段

这些能力放到第三阶段，当前不作为阻塞项：

- 自动调用外部 AI 完成代码。
- 自动把 task prompt 发送给写代码 AI。
- 自动接管真实项目里的浏览器操作。
- 自动截图对比 Demo 和真实页面。
- 自动根据失败结果循环修复到完全通过。
- 复杂权限托管和跨工具授权自动化。

## 第二阶段通过标准

满足以下条件即可认为第二阶段可用：

- 系统 AI 不依赖聊天记忆，也能通过文档和任务队列知道下一步。
- 写代码 AI 每次只拿到一个 task prompt，并能按协议回报。
- review 能发现报告缺失和 `allowedFiles` 越界改动。
- 失败后能生成修复任务，而不是让人重新口头解释。
- 问题能进入问题池，并能在总验收里体现。
- 可复用经验能写回 `08-项目实例` 和规则库。
- 人可以通过前端看到当前跑到哪一步、下一步该做什么。

## 提交前检查

- 不提交 `.obsidian/workspace.json`、本地运行记录里的临时调试文件和 `dist`。
- 确认文档链接存在。
- 控制台代码变更后运行：

```bash
cd apps/delivery-console
pnpm run build
node --check runner/server.mjs
```
