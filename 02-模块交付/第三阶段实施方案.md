# 第三阶段实施方案

> 用途：定义“自动执行器”怎么从第二阶段的人工复制粘贴，演进到系统 AI 自动调度写代码 AI、检查结果、生成修复任务和沉淀规则。第三阶段仍然要保持前端轻量，重心在 runner、协议和任务闭环。

## 阶段目标

第三阶段要解决的是“减少人工搬运”，不是让一个 AI 一次性写完整模块。

目标流程：

```text
一次性给齐资料和项目路径
  ↓
系统 AI 生成 design 和 task queue
  ↓
runner 锁定当前 task
  ↓
runner 调用写代码 AI 执行单 task
  ↓
写代码 AI 返回完成报告
  ↓
系统 AI / runner review 报告、git diff 和验证结果
  ↓
通过则派发下一个 task
  ↓
不通过则生成 fix task
  ↓
达到完成或修复上限后总验收和知识沉淀
```

## 边界原则

- 前端只展示状态、问题、当前任务、日志和人工确认入口。
- runner 负责文件读写、命令验收、git diff、任务锁和结果落盘。
- 系统 AI 负责 design、拆任务、review 结论和修复策略。
- 写代码 AI 只负责当前 task，不读取整个模块执行包，不越界。
- 第三阶段可以自动调用 AI，但不能绕过第二阶段的任务队列、`allowedFiles` 和 review。
- 不能自动提交 git、不能自动 push、不能执行任意命令。
- 涉及删除、重置、迁移大量文件、修改权限配置时必须进入人工确认。

## 最小架构

```text
apps/delivery-console
  ├─ 前端：展示任务队列、当前步骤、问题池、运行日志
  └─ runner：本地编排器
       ├─ 读取知识库和真实项目
       ├─ 生成 design / task queue / task prompt
       ├─ 调用 AI adapter
       ├─ 收集写代码 AI 报告
       ├─ 检查 git diff 和 allowedFiles
       ├─ 运行受控验证命令
       ├─ 生成 fix task
       └─ 写入验收和规则沉淀
```

AI adapter 必须是可替换的。系统不把某个 AI 平台写死在业务流程里。

## 执行器状态机

| 状态 | 含义 | 下一步 |
| --- | --- | --- |
| `idle` | 未开始自动执行 | 等待用户点击开始 |
| `planning` | 生成或读取 design 和任务队列 | 进入 `dispatching` |
| `dispatching` | 锁定当前 task，生成单任务输入 | 进入 `coding` |
| `coding` | 写代码 AI 执行当前 task | 进入 `reviewing` |
| `reviewing` | 系统 AI / runner 检查报告、diff、验证结果 | 通过进入 `dispatching`，失败进入 `fixing` |
| `fixing` | 生成并派发修复任务 | 进入 `coding` |
| `validating` | 全部任务后的命令、页面和样式验收 | 进入 `finalizing` 或 `fixing` |
| `finalizing` | 生成总验收和知识沉淀 | 完成 |
| `blocked` | P0 或权限问题，必须人工处理 | 等待人工确认 |

## 自动化权限

第三阶段开始自动执行前，必须形成一次性授权包：

```json
{
  "projectPath": "真实项目路径",
  "moduleName": "模块名称",
  "allowedWriteRoots": ["src/features/example/**"],
  "allowedCommands": ["typecheck", "lint", "build"],
  "maxRepairRounds": 3,
  "allowBrowserSmokeTest": false,
  "allowGitCommit": false,
  "allowGitPush": false
}
```

默认策略：

- `allowGitCommit` 固定为 `false`，提交由人执行。
- `allowGitPush` 固定为 `false`，推送由人执行。
- `allowedCommands` 只允许目标项目 `package.json` 里存在的脚本。
- `maxRepairRounds` 达到上限后停止，生成问题汇总给人确认。
- `allowedWriteRoots` 必须来自 task queue，不能由写代码 AI 自己扩大。

## AI adapter 输入输出

### 输入

AI adapter 每次只接收：

- 当前 task prompt。
- 写代码 AI 单任务协议。
- 任务要求的必要上下文。
- 真实项目中当前 task 允许读取的相关文件摘要。

### 输出

AI adapter 必须返回：

```text
任务编号：
实际改动文件：
完成内容：
运行命令和结果：
未完成内容：
新增问题：
建议下一步：
```

如果 AI provider 返回内容不符合格式，系统必须把它当成 review 不通过，生成修复或重新执行任务，而不是猜测成功。

## 检查策略

每个 task 完成后至少检查：

- 任务完成报告是否完整。
- 派发后新增 git 改动是否只落在 `allowedFiles`。
- 是否出现明显禁用模式，例如 `window.location.reload()`、页面直接 `fetch`、硬编码 mock。
- 是否需要运行 `typecheck`、`lint`、`build`。
- 是否新增 P0/P1/P2 问题。

第三阶段可以增加页面点测，但页面点测失败不能直接让 AI 乱改。必须先沉淀成问题，再生成修复任务。

## 修复循环

修复循环必须有限制：

```text
task 执行失败
  ↓
生成 fix task
  ↓
写代码 AI 修复
  ↓
系统 review
  ↓
仍失败且未超过 maxRepairRounds -> 再生成 fix task
  ↓
超过上限 -> blocked，生成问题清单
```

修复任务必须绑定具体问题，禁止“重写整个模块”。

## 分块实施

### 第三阶段 A：自动执行器干跑模式

状态：已实现。

目标：不调用外部 AI，只根据任务队列自动推进状态，生成“将要派发什么、将要检查什么”的运行计划。

验收：

- 能选择模块和任务队列。
- 能展示自动执行顺序。
- 能模拟 task -> review -> fix -> next 的状态变化。
- 不写真实项目。

### 第三阶段 B：AI adapter 抽象

状态：基础抽象已实现，已支持 `manual`、`mock` 和本地 `command` provider；平台化外部 AI provider 未接入。

目标：增加 provider 无关接口，先支持手工粘贴模式、mock provider 和可配置本地命令 provider。

验收：

- runner 里有统一 AI adapter 入口。已实现。
- 不在代码里写死密钥。已实现。
- provider 不可用时能降级为手工模式。已实现。
- 真实外部 AI provider 接入前，`manual` 是默认模式，`mock` 只用于测试调度链路。已实现。
- 本地 `command` provider 需要显式配置可信命令，prompt 通过 stdin 输入，并继续受单任务、执行锁、`allowedFiles` 和 review 约束。已实现。

### 第三阶段 C：真实单任务自动执行

状态：受控执行骨架已实现，可通过本地 `command` provider 接入写代码 AI；平台化外部 AI provider 未接入。

目标：在明确授权下，自动把当前 task 交给写代码 AI，并等待报告。

验收：

- 每次只执行一个 task。已实现。
- 每个 task 都记录输入、输出、review 和 git diff。已实现。
- 失败能生成 fix task。基于现有系统 review 已实现。
- 有执行锁和修复轮次上限。已实现。
- 不自动 commit / push，不无限循环修复。已实现。

### 第三阶段 D：轻量页面点测

状态：轻量 smoke test 已实现，截图对比和完整 E2E 未接入。

目标：对已完成模块做最小页面 smoke test，不做重型视觉平台。

验收：

- 能打开目标页面。已实现，runner 根据任务包里的页面 URL 请求页面。
- 能检查标题、主要控件、空态、loading、表格/列表是否存在。已实现轻量版：通过点测关键词检查页面文本是否存在。
- 能发现常见错误文本。已实现，例如 `Cannot read properties`、`ReferenceError`、`TypeError`、`Internal Server Error`。
- 点测结果写入知识库。已实现，生成 `{模块名}/页面点测-{时间戳}.auto.md`。
- 截图对比、真实浏览器交互和视觉回归未实现，后续单独扩展，避免前端控制台过重。

### 第三阶段 E：总验收和规则自动沉淀

状态：基础版已实现，规则仍需人工确认后进入通用规则库。

目标：把自动执行结果转成验收报告、问题清单和规则建议。

验收：

- 总验收能说明完成、警告、阻断。已实现，会检查任务队列、review、命令验收、页面点测、知识沉淀和问题池。
- 规则建议能回写到项目实例，人工确认后再进入通用规则库。已实现，生成 `{模块名}/规则沉淀候选.auto.md`。
- 不自动写入 `11-规则库`。已实现，避免未确认规则污染通用规则。

### 第三阶段 F：执行决策面板

状态：已实现。

目标：让系统 AI 的调度判断可见，避免用户只看到一堆状态卡但不知道下一步该做什么。

验收：

- 能说明当前阶段。已实现，例如资料补齐、生成设计、派发任务、等待报告、修复优先、命令验收、页面点测、知识沉淀、总验收、交付完成。
- 能说明下一步动作和责任方。已实现，区分用户、系统 AI、写代码 AI、测试和无需处理。
- 能说明判断依据。已实现，展示为什么现在要派任务、等报告、修复或验收。
- 能说明人要做什么、系统做什么。已实现。
- 能显示是否建议自动推进。已实现，只展示判断，不绕过任务队列和授权。

### 第三阶段 G：自动推进一步

状态：已实现。

目标：把系统 AI 的调度判断落成 runner 的最小执行单元，让用户不需要手动按一串按钮，但也不让系统无限循环。

验收：

- 每次只执行一个动作。已实现，动作包括生成任务队列、派发任务、受控执行、命令验收、页面点测、知识库写入和总验收。
- 遇到 assigned、blocked、命令失败、页面点测失败或缺少授权时停止。已实现。
- 不跳过写代码 AI 报告和系统 review。已实现。
- 前端只展示最近一次自动推进结果，不承担核心调度逻辑。已实现。

### 第三阶段 H：自动运行到暂停点

状态：已实现。

目标：在用户一次性给齐资料和授权后，由 runner 连续推进多个安全小步，直到遇到需要人处理的位置。

验收：

- 复用“自动推进一步”，不另写一套流程。已实现。
- 默认最多 20 步，避免无限循环。已实现。
- 遇到等待报告、阻断、命令失败、页面点测失败、缺少授权或交付完成即暂停。已实现。
- 返回每个小步的 action、status、summary 和 nextAction，方便前端展示进度。已实现。

## 第三阶段不做

- 不做复杂低代码平台。
- 不做完整需求管理系统。
- 不做多人协同权限系统。
- 不做自动 commit / push。
- 不做无限循环修复。
- 不做重型视觉回归平台。

## 成功标准

第三阶段成功不是“前端看起来很复杂”，而是：

- 人一次性提供资料后，系统能自己知道下一步。
- 写代码 AI 每次只处理一个小任务。
- 系统能检查写代码 AI 是否越界。
- 失败能自动转成修复任务或问题卡片。
- 长任务不会因为聊天上下文变长而迷路。
- 所有经验最终沉淀在知识库，而不是散落在对话里。
