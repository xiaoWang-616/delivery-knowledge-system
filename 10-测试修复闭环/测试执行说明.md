# 测试执行说明

> 用途：一轮开发完成后，指导 AI 或人工按固定顺序测试模块。

## 1. 测试前准备

| 检查项 | 说明 | 状态 |
| --- | --- | --- |
| 项目能启动 | 本地开发服务可以访问 |  |
| 测试账号可用 | 权限覆盖当前模块 |  |
| 测试数据可用 | 至少有正常数据和空数据场景 |  |
| 接口环境明确 | local/test/staging/prod |  |
| Demo/设计图可访问 | 用于样式验收 |  |
| 本轮改动清楚 | 知道这次改了哪些文件和功能 |  |

## 2. 推荐测试顺序

1. 代码检查。
2. 页面打开。
3. 核心功能。
4. 接口请求。
5. 样式对齐。
6. 性能风险。
7. 异常和空态。
8. 回归关联功能。

| 场景 | 检查点 | 结果 | 问题编号 |

## 5. 接口测试

打开浏览器 Network，检查：

| 检查项 | 通过标准 | 结果 | 问题编号 |
| --- | --- | --- | --- |
| 请求路径 | 符合接口文档 |  |  |
| 请求方法 | GET/POST/PUT/DELETE 正确 |  |  |
| 请求参数 | 不传未定义字段 |  |  |
| 分页参数 | 使用服务端分页 |  |  |
| 搜索请求 | debounce 或手动触发，不每字乱请求 |  |  |
| 写后刷新 | 只刷新相关数据 |  |  |
| 重复请求 | 没有无法解释的重复请求 |  |  |
| 无效请求 | 没有无意义 healthz 或旧桥接请求 |  |  |
