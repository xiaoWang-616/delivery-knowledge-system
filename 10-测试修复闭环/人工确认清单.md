# 人工确认清单

> 用途：把需要人判断的问题集中出来，方便转给产品、后端、设计或测试。

## 1. 确认原则

- 不让 AI 猜核心业务。
- 不让 AI 猜接口字段。
- 不让 AI 猜状态流转、权限、金额、删除、停用等高风险规则。
- 人工只需要回答清楚问题，不需要写技术方案。

## 2. 产品确认

| 编号 | 问题 | 背景 | 选项或期望答案 | 结论 | 确认人 |
| --- | --- | --- | --- | --- | --- |
|  |  |  |  |  |  |

适合产品确认的问题：

- 页面字段是否必须展示。
- 某个按钮什么条件下显示。
- 状态流转规则。
- 空态、异常态文案。
- 操作失败时用户应该看到什么。

## 3. 后端确认

| 编号 | 接口 | 问题 | 当前证据 | 期望后端答复 | 结论 |
| --- | --- | --- | --- | --- | --- |
|  |  |  |  |  |  |

适合后端确认的问题：

- 接口是否存在。
- 参数是否支持。
- 响应字段是否稳定。
- 错误码含义。
- 是否由后端做权限和业务阻断。

## 4. 设计确认

| 编号 | 页面 | 问题 | 当前差异 | 期望设计答复 | 结论 |
| --- | --- | --- | --- | --- | --- |
|  |  |  |  |  |  |

适合设计确认的问题：

- Demo 和旧项目不一致时，以哪个为准。
- 表格列太多时如何收缩。
- 弹窗宽度和表单布局。
- 空态、loading、tooltip 样式。

## 5. 测试确认

| 编号 | 场景 | 问题 | 需要测试提供 | 结论 |
| --- | --- | --- | --- | --- |
|  |  |  |  |  |

适合测试确认的问题：

- 是否有可用测试账号。
- 是否有边界测试数据。
- 哪些流程必须回归。
- 哪些问题本轮可以暂不处理。
